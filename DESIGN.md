## Problem Statement

現有 messenger 都不是為 AI agent 設計的。Telegram 有 Bot API 但 UX 是二等公民；WhatsApp 可以用 QR code 快速 onboard 但 agent 看起來像自己跟自己對話（workaround UX）；WeChat/LINE 商業化過重不可能開放。需要一個 **agent-first** 的 messenger，讓 agent 跟人一樣是 first-class citizen，同時保有 Signal 等級的隱私和 WhatsApp 等級的簡易 onboarding。

來源：Lman 在 OpenClaw 和 Hermes Agent 實際部署中遇到的痛點 — 每個 messenger 都要寫不同的 adapter，UX 都很彆扭。

## What Makes This Cool

1. **Agent = First-class citizen** — 不是 bot，是跟人同等的 entity，有自己的 identity、可以主動發訊息、可以跟其他 agent 對話
2. **QR onboarding for agents** — 掃一下就接上，不用 API key、不用註冊、不用寫 code
3. **TUI in chat** — Agent 可以送出互動式介面（按鈕、表格、卡片），不只是純文字
4. **E2E P2P** — 沒有公司能看到你跟 agent 的對話
5. **Agent Economy** — 長期 vision：agent 代表你付款、簽約、交易，錢包地址就是 identity

## Target Audience

**Primary**: Agent 開發者（OpenClaw/Hermes/Claude Code 生態系）— 需要一個 distribution channel 讓終端用戶能輕易使用他們的 agent。

**Secondary**: 技術早期採用者、crypto-native 用戶、隱私敏感用戶。

## Identity System

多重 identity，降低不同用戶群的 onboarding 摩擦：

| Identity Type | 用戶群 | Onboarding |
|---------------|--------|------------|
| 電話號碼 | 一般用戶（最低摩擦） | SMS 驗證 → Nostr keypair |
| 錢包地址 | Crypto-native | 簽名驗證 → Nostr keypair |
| Nostr pubkey | 技術用戶 | 直接用 |

所有 identity 最終都映射到 Nostr keypair。電話號碼通過 NIP-05 style 的 verification service 映射。

**Agent Identity**: 明確標示為 Agent（類似 Telegram verified bot badge），包含 capability manifest（這個 agent 能做什麼）。

## Constraints

- **Protocol**: Nostr（不自建）— 用 relay infra + NIP 加密
- **Client**: SwiftUI（iOS + Mac 共用）+ Tauri（Windows）
- **Agent SDK**: 極簡 — `npm install sigil-agent` 或 Python package，< 30 分鐘 onboarding
- **Open Source**: MIT license，protocol 層完全開放
- **No heavy server**: relay 是 commodity（用現有 Nostr relay 或自建輕量 relay）

## Premises

1. 現有 messenger 不會在 1-2 年內原生支援 agent-as-first-class-citizen
2. Nostr protocol 足夠成熟作為 agent messaging 基礎層
3. Agent 開發者願意讓 agent 說 Nostr（需要簡單的 bridge/SDK）
4. Identity 支援電話號碼 + 錢包地址 + Nostr pubkey 三種方式
5. 價值在 open protocol + 生態，不在封閉平台

## Approaches Considered

### Approach A: Fork Damus
Fork Damus iOS client，加 Agent tab。最快但 fork maintenance debt 重，Windows 要另做。Completeness: 6/10。

### Approach B: Custom Protocol + Tauri
自建 JSON-over-WebSocket protocol + libsignal E2E。完全控制但從零建一切，8-12 weeks。Completeness: 9/10。

### Approach C: Nostr Protocol + Custom Swift Client ✅ CHOSEN
用 Nostr 作為 protocol 層，自己寫 SwiftUI client（不 fork Damus）。定義 Agent NIP 標準。Agent bridge 讓現有 agent framework 能收發 Nostr message。Completeness: 8/10。

## Open Questions

1. **Phone verification service** — 需要一個中心化服務做 SMS → Nostr keypair mapping。這跟 P2P 精神有張力。可能的解法：federated verification（多個 provider 競爭）。
2. **TUI format 標準化** — 要自己定義還是用現有標準（如 Adaptive Cards）？
3. **Relay selection** — agent 和 user 用同一個 relay 還是分開？performance 考量。
4. **Codename** — 暫定 "Sigil"（符文，代表 agent 的標記），需要確認是否好用。
5. **License** — MIT 還是 AGPL（防止閉源 fork）？

## Success Criteria

1. **Agent onboarding < 2 分鐘**：掃 QR → agent connected → 第一個回覆
2. **10 個 agent 開發者接上** SDK 並跑通
3. **3 platform** client 可用（iOS + Mac + Windows）
4. **NIP proposal** 提交到 Nostr 社群獲得討論

## What I noticed about how you think

- 你從 OpenClaw 的實戰痛點出發，不是從技術 spec 出發。「WhatsApp 的 UX 怪怪的，看起來是自己跟自己對話」— 這是用戶體驗的觀察，不是工程師的抱怨。
- 你說「全部都要」但不是貪心 — 你清楚 Agent 協作 + TUI + E2E 三者加起來才是完整的差異化，少一個就不夠。這是產品直覺。
- 選 Agent Economy 作為 10x vision 而不是 marketplace — 你想的是基礎設施層，不是應用層。這跟 IrisGo 的定位（AI OS layer）一致。
- 堅持電話號碼 identity — 你知道 crypto-only identity 會把 99% 的人擋在門外。這是對 UX 摩擦力的敏感度。
